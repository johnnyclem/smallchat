smallchat **0.5.0** is out. This release closes out the confidence-tiered dispatch work that started in 0.4.0, lines up every workspace package on the same version, and ships our first dedicated integration guide — for [LoomMCP](https://muhnehh.github.io/loom-mcp/), the symbol-graph MCP server that pairs unreasonably well with semantic dispatch.

If you're new here: smallchat compiles your MCP tools into a dispatch table. The LLM expresses *intent* in natural language; the runtime resolves it to the right tool, semantically and deterministically, in microseconds. No prompt stuffing. No selection lottery. The model never sees your tool list.

## What's in 0.5.0

### Synchronized release across every package

Previous releases had drifted: `@smallchat/core` was at 0.4.0 while the satellites — `@smallchat/react`, `@smallchat/nextjs`, `@smallchat/testing`, `@smallchat/playground`, `@smallchat/docs`, `@smallchat/examples`, and the `smallchat-vscode` extension — were stuck back at 0.3.0. As of 0.5.0 they all march in lockstep. Pin one, pin them all.

```bash
npm install @smallchat/core@0.5.0
npm install @smallchat/react@0.5.0
npm install @smallchat/nextjs@0.5.0
```

### Runtime version metadata refreshed

The MCP server, the channel server, the MCP client's `clientInfo`, the REPL banner, and the compiled artifact format all now report `0.5.0`. Existing 0.3.0/0.4.0 artifacts continue to load — recompile only if you want the metadata to refresh.

```bash
$ npx @smallchat/core repl tools.toolkit.json

Type an intent to resolve, or :help for commands.
```

### LoomMCP integration guide

This is the part I'm most excited about.

[LoomMCP](https://muhnehh.github.io/loom-mcp/) is an MCP server that turns a codebase into a queryable symbol graph. Instead of paying tokens to re-read whole files, an agent calls one of LoomMCP's 17 tools — `loom_get_topology`, `loom_focus`, `loom_search_refs`, and friends — and gets back exactly the function, class, or reference set it asked for. The project reports an average **97% reduction in tokens** spent reading code.

LoomMCP and smallchat solve adjacent problems:

| Layer | Problem | Tool |
|---|---|---|
| **Retrieval** | "Don't read whole files; fetch the right symbols." | LoomMCP |
| **Dispatch** | "Don't dump 17 tool schemas in the prompt; route the intent." | smallchat |

Wire them together and the agent stops paying twice — once for tool selection, once for source code.

Here's the punchline. Point smallchat at LoomMCP:

```json title="loom.mcp.json"
{
  "mcpServers": {
    "loom": {
      "command": "npx",
      "args": ["@loom-mcp/server"]
    }
  }
}
```

```bash
npx @smallchat/core compile --source ./loom.mcp.json --output loom.toolkit.json
# Compiling tools... ✓ 17 tools from 1 provider embedded.
```

And dispatch by intent — the agent never sees the tool names:

```typescript
import { ToolRuntime, LocalEmbedder, MemoryVectorIndex } from '@smallchat/core';

const runtime = new ToolRuntime({
  embedder: new LocalEmbedder(),
  vectorIndex: new MemoryVectorIndex(),
});
await runtime.load('./loom.toolkit.json');

await runtime.dispatch('scan the src directory',         { path: 'src/' });
await runtime.dispatch('focus on the loginUser function', { symbol: 'src/auth.ts::loginUser' });
await runtime.dispatch('find every caller of loginUser',  { symbol: 'loginUser' });
```

The full walkthrough — including how to front the whole thing as a single MCP endpoint with `smallchat serve` — is on the new [LoomMCP integration page](/docs/integrations/loom-mcp).

## Carried forward from 0.4.0

If you skipped 0.4.0, here's what landed there and is now stable in 0.5.0.

### Confidence-tiered dispatch (Pillar 1)

Every dispatch returns a confidence tier — `EXACT`, `HIGH`, `MEDIUM`, `LOW`, or `NONE` — and the runtime branches on it. MEDIUM triggers verification. LOW triggers decomposition. NONE triggers refinement. Tiers are actionable, not just diagnostic.

### Resolution proof

Every dispatch carries a serializable `ResolutionProof` trace documenting *exactly* why a tool was chosen, with per-step timing. Audit-friendly by default.

### Pre-flight verification (Pillar 2)

A `respondsToSelector:` gate sits between resolution and execution. Three progressive strategies — schema validation, keyword overlap, and an optional LLM micro-check — catch misroutes before they hit your tool's implementation.

### Intent decomposition (Pillar 3)

LOW-confidence dispatches become a `doesNotUnderstand:` handler that breaks the complex intent into sub-intents and dispatches them individually with dependency resolution. "Find me a flight to NYC and book the cheapest hotel near JFK" becomes two dispatches, executed in order.

### Refinement protocol (Pillar 4)

NONE-confidence dispatches return a structured `tool_refinement_needed` result type — a `forwardInvocation:` dialogue your agent can show to the user to narrow the intent, with concrete options drawn from the nearest matches.

### Observation & adaptation (Pillar 5)

A KVO-inspired dispatch observer tracks correction signals, schema rejections, and known-bad intent+tool pairs, then adapts per-tool-class thresholds in real time. Your dispatch table gets sharper with use.

### Pluggable LLM client + `--strict` mode

All LLM-powered features (verification, decomposition, refinement) sit behind a single `LLMClient` interface and degrade gracefully without one. The new `--strict` compile flag raises thresholds, enables verification on every dispatch, and treats ambiguity as an error — turn it on for production.

## Upgrading

Inside an existing project:

```bash
npm install @smallchat/core@0.5.0
```

If you use any satellites, bump them in lockstep:

```bash
npm install \
  @smallchat/core@0.5.0 \
  @smallchat/react@0.5.0 \
  @smallchat/nextjs@0.5.0 \
  @smallchat/testing@0.5.0
```

Existing compiled artifacts from 0.3.0 and 0.4.0 still load. Recompile if you want the artifact metadata to read `0.5.0`, or if you want to opt into the expanded collision firewall and adaptive thresholds.

## What's next

Two things on the near horizon:

- **More integration guides** — LoomMCP is the first; expect short, opinionated walkthroughs for the MCP servers that pair best with semantic dispatch.
- **Streaming refinement UX** — turning the `tool_refinement_needed` result into a first-class component in `@smallchat/react` so agents can ask the user to narrow an intent without bespoke plumbing.
